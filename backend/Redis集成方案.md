## 概述

BiliNote 项目通过引入 Redis，解决了三大核心痛点：
1. **任务状态查询性能差**：文件 I/O 频繁，响应慢
2. **缓存无过期机制**：磁盘空间浪费，需要手动清理
3. **缓存查询效率低**：文件系统查询慢，用户体验差

本文档通过实际场景和代码示例，详细说明 Redis 如何解决这些问题。

## 痛点1：任务状态查询性能差

### 🔴 问题描述

**优化前的情况**：
- 前端每秒轮询读取 `{task_id}.status.json` 文件
- 每次查询需要磁盘 I/O 操作
- 平均响应时间：5-10ms
- 高并发时磁盘成为瓶颈

### 📋 实际场景举例

```
用户场景：用户提交一个视频处理任务

优化前的流程：
┌─────────────────────────────────────────────────────────┐
│ 1. 用户点击"生成笔记"                                    │
│ 2. 前端每秒发送请求查询状态                              │
│ 3. 后端每次都要：                                        │
│    - 打开文件 note_results/abc123.status.json           │
│    - 读取文件内容                                        │
│    - 解析 JSON                                          │
│    - 返回状态                                           │
│ 4. 一个任务处理 60 秒，就要读取文件 60 次                │
│ 5. 10 个用户同时处理，就是 600 次文件 I/O               │
└─────────────────────────────────────────────────────────┘

问题：
❌ 磁盘 I/O 频繁，系统负载高
❌ 响应时间不稳定（5-10ms）
❌ 用户感觉状态更新有延迟
❌ 高并发时性能下降明显
```

### ✅ Redis 解决方案

**优化后的代码**：
```python
# backend/app/services/note.py
def _update_status(self, task_id, status, message=None):
    """使用 Redis Hash 存储任务状态"""
    from app.utils.redis_client import RedisManager
    
    # 写入 Redis（内存操作，毫秒级）
    redis_manager = RedisManager(db=0)
    redis_key = f"task:{task_id}"
    data = {
        "status": status.value,
        "message": message,
        "updated_at": str(time.time())
    }
    # 写入 Redis，24小时自动过期
    redis_manager.hset(redis_key, data, ttl=86400)
```

## 痛点2：缓存无过期机制

### 🔴 问题描述

**优化前的情况**：
- 音频文件、转写结果、Markdown 永久保存在 `note_results/` 目录
- 目录持续膨胀，占用大量磁盘空间
- 需要手动清理，运维成本高
- 不知道哪些文件还有用，哪些可以删除

### ✅ Redis 解决方案

**转写结果缓存（7天过期）**：
```python
# backend/app/services/note.py
def _transcribe_audio(self, audio_file, transcript_cache_file, status_phase):
    """转写音频，使用 Redis 缓存，7天自动过期"""
    from app.utils.redis_client import RedisManager
    
    cache_key = f"transcript:{platform}_{video_id}_{quality}"
    redis_manager = RedisManager(db=1)
    
    # 优先从 Redis 读取
    cached_data = redis_manager.get(cache_key)
    if cached_data:
        logger.info(f"✅ 从 Redis 读取转写缓存")
        return json.loads(cached_data)
    
    # 执行转写
    transcript = self.transcriber.transcript(file_path=audio_file)
    
    # 写入 Redis，设置 7 天过期
    redis_manager.set(
        cache_key,
        json.dumps(asdict(transcript), ensure_ascii=False),
        ttl=604800  # 7天后自动删除
    )
    
    return transcript
```

## 痛点3：缓存查询效率低

### 🔴 问题描述

**优化前的情况**：
- 相同视频重复处理时，需要遍历文件系统查找缓存
- 文件 I/O 查询慢（10-50ms）
- 大文件读取耗时长
- 缓存命中时仍需较长时间

### ✅ Redis 解决方案

**三级缓存策略**：
```python
def _transcribe_audio(self, audio_file, transcript_cache_file, status_phase):
    """三级缓存：Redis → 文件 → 执行转写"""
    cache_key = f"transcript:{platform}_{video_id}_{quality}"
    redis_manager = RedisManager(db=1)
    
    # 第一级：Redis 缓存（最快，1-2ms）
    if redis_manager.available:
        cached_data = redis_manager.get(cache_key)
        if cached_data:
            logger.info(f"✅ 从 Redis 读取转写缓存（1-2ms）")
            return json.loads(cached_data)
    
    # 第二级：文件缓存（较快，10-50ms）
    if transcript_cache_file.exists():
        logger.info(f"📁 从文件读取转写缓存（10-50ms）")
        data = json.loads(transcript_cache_file.read_text())
        
        # 回写到 Redis，下次更快
        if redis_manager.available:
            redis_manager.set(cache_key, json.dumps(data), ttl=604800)
        
        return TranscriptResult(**data)
    
    # 第三级：执行转写（最慢，30-120秒）
    logger.info(f"🎤 开始转写音频（30-120秒）")
    transcript = self.transcriber.transcript(file_path=audio_file)
    
    # 写入 Redis 和文件（双写）
    redis_manager.set(cache_key, json.dumps(asdict(transcript)), ttl=604800)
    transcript_cache_file.write_text(json.dumps(asdict(transcript)))
    
    return transcript
```
