# 合集视频并行处理方案

## 问题描述

当前合集处理流程是串行的：
```
下载视频1 → 下载视频2 → 下载视频3 → 转写1 → GPT1 → 转写2 → GPT2 → 转写3 → GPT3
```

这导致：
- 下载时 CPU/GPU 空闲
- 转写时网络空闲
- 总耗时 = 所有步骤时间之和

## 优化目标

实现流水线并行处理：
```
下载1 → 下载2 → 下载3 → ...
  ↓       ↓       ↓
转写1 → 转写2 → 转写3 → ...
  ↓       ↓       ↓
GPT1  → GPT2  → GPT3  → ...
```

**预期收益**：
- 总耗时 ≈ max(下载总时间, 转写总时间, GPT总时间)
- 资源利用率提升 200-300%

---

## 方案一：生产者-消费者模式（推荐）

### 核心思路

使用 Python `asyncio` + `Queue` 实现三级流水线：

1. **下载队列**：存放待下载的视频 URL
2. **转写队列**：存放已下载的音频文件
3. **总结队列**：存放已转写的文本

### 架构设计

```python

# 三个异步工作线程
async def download_worker(video_urls, download_queue):
    for url in video_urls:
        audio = await download(url)
        await download_queue.put(audio)

async def transcribe_worker(download_queue, transcribe_queue):
    while True:
        audio = await download_queue.get()
        transcript = await transcribe(audio)
        await transcribe_queue.put((audio, transcript))

async def gpt_worker(transcribe_queue, results):
    while True:
        audio, transcript = await transcribe_queue.get()
        markdown = await gpt_summarize(transcript)
        results.append(markdown)
```

### 优点
- 最大化并行度
- 资源利用率高（网络、CPU、GPU 同时工作）
- 实时进度更新

### 缺点
- 需要改造为异步代码
- 实现复杂度较高

## 方案二：线程池并行（简单）

### 核心思路

使用 `ThreadPoolExecutor` 实现边下载边处理：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def process_playlist_parallel(video_urls):
    results = []
    
    with ThreadPoolExecutor(max_workers=3) as executor:
        # 提交所有下载任务
        download_futures = {
            executor.submit(download_and_process, url): idx
            for idx, url in enumerate(video_urls)
        }
        
        # 谁先完成就先处理谁
        for future in as_completed(download_futures):
            result = future.result()
            results.append(result)
    
    return results

def download_and_process(url):
    # 下载
    audio = download(url)
    # 转写
    transcript = transcribe(audio)
    # GPT
    markdown = gpt_summarize(transcript)
    return markdown
```

### 优点
- 实现简单，改动小
- 不需要异步改造

### 缺点
- 并行度有限（受线程数限制）
- 无法实现真正的流水线

## 方案三：混合方案（推荐实现）

### 核心思路

分两阶段并行：
1. **阶段1**：并行下载所有视频
2. **阶段2**：并行处理（转写 + GPT）

```python
async def process_playlist_hybrid(video_urls):
    # 阶段1：并行下载
    download_tasks = [download_async(url) for url in video_urls]
    audios = await asyncio.gather(*download_tasks)
    
    # 阶段2：并行处理
    process_tasks = [
        process_single_video(audio) 
        for audio in audios
    ]
    results = await asyncio.gather(*process_tasks)
    
    return results

async def process_single_video(audio):
    # 转写（可能占用 GPU）
    transcript = await transcribe_async(audio)
    # GPT（网络 IO）
    markdown = await gpt_async(transcript)
    return markdown
```

### 优点
- 实现难度适中
- 下载阶段完全并行
- 处理阶段也能并行（如果资源允许）

### 缺点
- 不是真正的流水线
- 需要等所有下载完成才开始处理

## 方案四：渐进式优化（最简单）

### 核心思路

保持现有代码结构，只优化下载部分：

```python
def process_playlist_progressive(video_urls):
    results = []
    downloaded = []
    
    # 启动后台下载线程
    download_thread = threading.Thread(
        target=download_all_in_background,
        args=(video_urls, downloaded)
    )
    download_thread.start()
    
    # 主线程：边等待边处理
    processed_count = 0
    while processed_count < len(video_urls):
        # 检查是否有新下载完成的
        if len(downloaded) > processed_count:
            audio = downloaded[processed_count]
            
            # 处理这个视频
            transcript = transcribe(audio)
            markdown = gpt_summarize(transcript)
            results.append(markdown)
            
            processed_count += 1
        else:
            time.sleep(1)  # 等待下载
    
    download_thread.join()
    return results
```

### 优点
- 改动最小
- 向后兼容
- 立即见效

### 缺点
- 并行度有限
- 只优化了下载环节

## 实现细节

### 1. 进度更新

需要更细粒度的状态：

```python
{
    "status": "processing_playlist",
    "total_videos": 10,
    "downloaded": 5,      # 已下载
    "transcribed": 3,     # 已转写
    "summarized": 2,      # 已总结
    "current_action": "正在下载第6个视频..."
}
```

### 2. 错误处理

某个视频失败不应影响其他视频：

```python
try:
    result = process_video(url)
    results.append(result)
except Exception as e:
    logger.error(f"视频处理失败: {url}, 错误: {e}")
    results.append({
        "error": str(e),
        "title": "处理失败"
    })
```

### 3. 资源限制

避免同时处理太多视频导致内存/GPU 爆炸：

```python
# 限制并发数
MAX_CONCURRENT_DOWNLOADS = 3
MAX_CONCURRENT_TRANSCRIPTIONS = 2  # GPU 资源有限
MAX_CONCURRENT_GPT_CALLS = 5
```

### 4. 顺序保证

最终合并时需要按原始顺序：

```python
# 使用字典保存，key 是索引
results = {}
results[0] = markdown_1
results[2] = markdown_3
results[1] = markdown_2

# 按顺序合并
final_markdown = "\n\n".join([
    results[i] for i in sorted(results.keys())
])
```

## 性能对比

### 示例：10 个视频的合集

**当前串行方案**：
- 下载：10 × 30秒 = 300秒
- 转写：10 × 60秒 = 600秒
- GPT：10 × 20秒 = 200秒
- **总计：1100秒（18分钟）**

**方案四（渐进式）**：
- 下载和处理并行
- **总计：约 700秒（12分钟）**
- 提升：36%

**方案三（混合）**：
- 下载并行：30秒
- 处理并行（假设 3 个并发）：(600+200)/3 = 267秒
- **总计：约 300秒（5分钟）**
- 提升：73%

**方案一（完整流水线）**：
- 最慢环节决定速度：max(300, 600, 200) = 600秒
- **总计：约 600秒（10分钟）**
- 提升：45%

## 注意事项

1. **GPU 资源**：转写可能占用 GPU，不能无限并行
2. **API 限流**：GPT API 可能有速率限制
3. **内存管理**：避免同时加载太多音频文件
4. **磁盘 IO**：并发写入可能成为瓶颈
5. **网络带宽**：下载并发数不宜过高
