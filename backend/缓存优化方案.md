# 视频缓存优化方案

## 问题描述

当前系统每次处理同一个视频时都会重新下载，造成资源浪费。需要实现缓存机制，避免重复下载。

## 解决方案

### 核心思路

采用**混合缓存方案**：
- 音频文件：基于视频 ID 的文件系统缓存
- 转写结果：数据库 + 文件存储
- GPT 结果：数据库存储
- 完整性校验：多层次验证机制

---

## 一、文件缓存策略

### 1.1 文件命名规则

```
data/
  audio/
    {platform}_{video_id}_{quality}.mp3
    
示例：
    bilibili_BV1234567890_medium.mp3
    youtube_dQw4w9WgXcQ_high.mp3
```

### 1.2 缓存查询流程

```
1. 生成缓存文件名
2. 检查文件是否存在
3. 校验文件完整性（多层次）
4. 如果有效，直接使用
5. 如果无效或不存在，重新下载
```

## 二、数据库设计

### 2.1 缓存表结构

```sql
CREATE TABLE video_cache (
    id INTEGER PRIMARY KEY,
    video_id TEXT NOT NULL,
    platform TEXT NOT NULL,
    quality TEXT NOT NULL,
    
    -- 文件信息
    audio_file_path TEXT NOT NULL,
    audio_file_size INTEGER NOT NULL,
    audio_file_hash TEXT,
    
    -- 音频元数据
    audio_duration REAL NOT NULL,
    audio_bit_rate INTEGER,
    audio_codec TEXT,
    
    -- 视频元信息
    title TEXT,
    author TEXT,
    
    -- 缓存管理
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_accessed_at TIMESTAMP,
    last_validated_at TIMESTAMP,
    validation_status TEXT DEFAULT 'valid',
    
    UNIQUE(video_id, platform, quality)
);

CREATE INDEX idx_video_cache_lookup ON video_cache(video_id, platform, quality);
```

### 2.2 转写缓存表

```sql
CREATE TABLE transcript_cache (
    id INTEGER PRIMARY KEY,
    video_id TEXT NOT NULL,
    platform TEXT NOT NULL,
    transcriber_type TEXT NOT NULL,
    transcript_file_path TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(video_id, platform, transcriber_type)
);
```

### 2.3 GPT 缓存表

```sql
CREATE TABLE summary_cache (
    id INTEGER PRIMARY KEY,
    video_id TEXT NOT NULL,
    platform TEXT NOT NULL,
    model_name TEXT NOT NULL,
    summary_text TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(video_id, platform, model_name)
);
```

## 三、文件完整性校验

### 3.1 多层次校验策略

#### 第一层：快速检查（必须）
- 文件是否存在
- 文件大小是否匹配（允许 10% 误差）
- 文件是否为空

#### 第二层：元数据校验（默认）
- 使用 ffprobe 读取音频元数据
- 校验时长（允许 2 秒误差）
- 校验编码格式
- 验证文件可播放性

#### 第三层：哈希校验（可选）
- 计算文件 MD5/SHA256
- 与数据库记录对比
- 用户可在设置中开启

### 3.2 校验级别

```python
# 三种校验级别
'quick'   # 快速：只检查文件存在和大小
'normal'  # 正常：快速检查 + 元数据校验（默认）
'strict'  # 严格：完整哈希校验
```

### 3.3 校验优化

- 如果 24 小时内已校验过，只做快速检查
- 定期后台任务校验所有缓存
- 校验失败自动重新下载

## 四、实现流程

### 4.1 下载流程（带缓存）

```
1. 解析视频 URL，获取 video_id
2. 生成缓存文件名
3. 查询数据库是否有缓存记录
4. 如果有缓存：
   a. 校验文件完整性
   b. 如果有效，返回缓存路径
   c. 如果无效，删除记录，继续下载
5. 如果无缓存：
   a. 下载音频文件
   b. 重命名为缓存文件名
   c. 提取音频元数据
   d. 保存到数据库
   e. 返回文件路径
```

### 4.2 转写流程（带缓存）

```
1. 检查 transcript_cache 表
2. 如果有缓存且转写器类型匹配：
   a. 读取缓存文件
   b. 返回转写结果
3. 如果无缓存：
   a. 执行转写
   b. 保存转写结果到文件
   c. 记录到数据库
   d. 返回转写结果
```

### 4.3 GPT 总结流程（带缓存）

```
1. 检查 summary_cache 表
2. 如果有缓存且模型匹配：
   a. 直接返回缓存的 markdown
3. 如果无缓存：
   a. 调用 GPT API
   b. 保存结果到数据库
   c. 返回 markdown
```

## 五、缓存管理

### 5.1 缓存清理策略

#### 自动清理
- 超过 30 天未访问的缓存
- 校验失败的损坏文件
- 磁盘空间不足时清理最旧的缓存

#### 手动清理
- 用户可在设置中清理所有缓存
- 可按平台、日期范围清理
- 清理前显示可释放的空间

### 5.2 缓存统计

```sql
-- 缓存命中率统计表
CREATE TABLE cache_stats (
    date DATE PRIMARY KEY,
    total_requests INTEGER DEFAULT 0,
    cache_hits INTEGER DEFAULT 0,
    cache_misses INTEGER DEFAULT 0,
    bytes_saved INTEGER DEFAULT 0
);
```

## 七、性能优化

### 7.1 预期收益

- **首次下载**：无变化
- **重复下载**：节省 100% 下载时间
- **重复转写**：节省 100% 转写时间（如果转写器相同）
- **重复总结**：节省 100% GPT 调用（如果模型相同）

### 7.2 示例场景

**场景 1：用户重新生成笔记（相同配置）**
- 下载：0 秒（缓存）
- 转写：0 秒（缓存）
- GPT：0 秒（缓存）
- 总耗时：< 1 秒

**场景 2：用户切换转写器**
- 下载：0 秒（缓存）
- 转写：需要重新转写
- GPT：需要重新总结
- 总耗时：转写时间 + GPT 时间

**场景 3：用户切换 GPT 模型**
- 下载：0 秒（缓存）
- 转写：0 秒（缓存）
- GPT：需要重新总结
- 总耗时：GPT 时间

## 九、注意事项

### 9.1 边界情况

- 视频被删除或下架：缓存仍然有效
- 视频内容更新：需要用户手动清理缓存
- 不同质量的音频：分别缓存
- 合集视频：每个视频单独缓存

### 9.2 安全性

- 定期校验缓存完整性
- 防止缓存污染（损坏文件）
- 磁盘空间监控

### 9.3 兼容性

- 现有代码改动最小化
- 向后兼容（旧的 task_id 文件不受影响）
- 渐进式迁移
